Adding Datto RMM to my MCP server

9 July 2026 · MCP · Python · Datto RMM
MCPPythonDatto RMMSecurity

The MCP server I built to search our ITGlue documentation started out doing one job: turning a pile of static documentation into something an agent could search. It worked well enough that the obvious next question was whether the same agent could also see what was actually happening on our clients' machines right now, not just what was written down about them. That meant bringing in Datto RMM, the remote monitoring and management tool the MSP I work at uses to keep an eye on client devices.

Mirroring Datto RMM alongside ITGlue

I followed the same pattern as the ITGlue sync: a Python layer that pulls sites, devices (with the hardware and software audit data flattened out) and open alerts from Datto's REST API, and writes them into the same local SQLite database, indexed for full-text search alongside the documentation. Datto's auth is a bit different to ITGlue's, an OAuth2 password grant against their token endpoint, authenticated with a Basic header using Datto's own public client credentials rather than anything secret, then the account's API key and secret as the username and password.

One decision carried straight over from a lesson I'd already learned the hard way on this project: Datto RMM has free-text UDF fields (user-defined fields technicians can put anything in), and I exclude them everywhere, from the device audit data, the site notes, all of it. I'd previously had to rip out an entire Autotask integration because a similar free-text field turned out to hold plaintext customer credentials, so this time I built the exclusion in from the start rather than discovering the problem later.

For a handful of things where the mirror would go stale too quickly to be useful, patch status, antivirus state, who's currently logged in, I added a few tools that skip the local database and call Datto's API directly. Everything else is served from the mirror, which keeps most tool calls fast and doesn't hammer Datto's rate limits every time someone asks a question.

Giving the agent a narrow way to actually do something

Everything up to that point was read-only, which felt like the right, cautious way to build a tool that a semi-autonomous agent has access to. But a lot of what makes an RMM genuinely useful is being able to run something on a device: restart a service, clear a cache, kick off a script. That's a different category of risk to a search tool. Handing an agent the ability to run arbitrary code on a client's endpoint isn't something I was willing to do just because it would be convenient.

So instead of letting the agent supply a component and its parameters, I built a fixed allowlist. Each entry maps a plain, friendly key to a specific, pre-vetted Datto component ID and a fixed set of variables, copied by hand from the Datto web UI (the component ID isn't something you can discover through the API at all, which made the allowlist feel less like a workaround and more like the only sane design). The agent picks a key from the list, nothing more. It never sees or sets a raw component ID or variable, so there's no way to smuggle something unvetted through. The whole feature also sits behind an opt-in flag that's off by default, so a deployment has to deliberately turn it on before these tools even exist to the agent.

The job itself runs asynchronously: submit it and get back a job ID, poll until Datto reports it's finished, then fetch the stdout and stderr. Wiring that up taught me more about designing a tool boundary than the read-only side did. With a search tool, the interesting design question is what to expose. With a write tool, the interesting question is what to refuse to expose, and building the allowlist first, before writing a single line of the tool itself, made that much easier to reason about.

A concrete example that ended up in the allowlist: we'd been chasing a bug on some Dell machines where a database file behind the CapabilityAccessManager service (the bit of Windows that tracks which apps have been granted access to things like the camera and microphone) would balloon in size over time, eating disk space and dragging the machine down until someone noticed and cleared it by hand. Rather than write a general-purpose "run this arbitrary PowerShell" tool, I added two narrow, single-purpose entries: one that checks the size of that database file and reports back, and a separate one that stops the service, clears the bloated file and restarts it. The agent can ask "does this device have the CapabilityAccessManager bloat issue" and, if the answer is yes, run the fix, but it can only ever run those two specific, pre-tested actions, never anything more general.

It's a small addition, but it's the part of this project I'm most pleased with. The server can now do a bit more than answer questions, on my terms, within limits I set in advance rather than ones an agent has to be trusted to respect on its own.
